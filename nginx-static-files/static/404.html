404 Page - Use Cases and Best Practices

General Use for "Not Found" Cases

A 404 error page is displayed when a user attempts to access a page that does not exist on the server. This typically occurs when:

  • The URL is typed incorrectly.
  • A resource has been moved or deleted without proper redirection.
  • A broken or outdated link leads to the website.

The purpose of a 404 page is to:

  • Inform the user that the requested resource could not be found.
  • Provide options or links to guide the user back to functional parts of the website.
  • Maintain a consistent branding and user experience.

404 Page for Single Page Applications (SPAs)

In Single Page Applications (SPAs), all navigation and content rendering is typically handled by JavaScript on the client side. This can cause a problem with 404 errors, as direct access to specific routes may bypass the SPA’s routing logic, leading to a "not found" error from the server.

To prevent this, the server should be configured to serve the main index.html file for any unknown routes. The SPA can then handle the routing internally. Here’s how a 404 page is used in SPAs:

  • If a user accesses a route (e.g., /about or /profile) directly, the server should return the main index.html.
  • The SPA's JavaScript router inspects the route and displays the correct content, or redirects to a custom "Page Not Found" component for invalid routes.

This approach ensures that:

  • All valid routes are handled correctly by the SPA.
  • Custom 404 logic is implemented within the application for invalid routes.

Example Configuration for SPAs

In the Nginx configuration, you can use the following directive to enable SPA routing: 

        location / {
            try_files $uri /index.html;
        }

This ensures that all routes are served the main index.html, allowing the SPA to handle navigation and 404 cases internally.
